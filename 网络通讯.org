* tcp <2021-08-31 Tue>
** 三次握手
- 第一次握手: 客户端发送syn包，等待服务器确认(SYN_SEND)
- 第二次握手: 服务器响应，发送syn包(SYN+ACK)
- 第三次握手: 客户端收到SYN+ACK，并返回服务器
** 四次挥手
- 第一次: 客户端通知服务端要断开
- 第二次: 服务端回应客户端
- 第三次: 服务端通知客户端要断开
- 第四次: 客户端回应服务端

** TODO 七层网络模型(OSI) <Open System Interconnection>
- 物理层(光纤 线路): 建立、维护、断开物理连接
- 数据链路层(以太网): 建立逻辑连接、进行硬件地址寻址、差错校验等功能
- 网络层(ip协议 IPV4/IPV6): 进行逻辑地址寻址，实现不同网络之间的路径选择
- 传输层(TCP/UDP): 定义传输数据的协议端口号，以及流控和差错校验
- 会话层(RPC): 建立、管理、终止会话
- 表示层(JPEG/ASCII): 数据的表示、安全、压缩
- 应用层(HTTP/HTTPS/SMTP/DNS): 网络服务与最终用户的一个接口

#+begin_center
物理层是硬件层 数据链路到传输是内核程序(操作系统内置程序)
#+end_center

* 端口 <2021-09-02 Thu>
http的缺省端口是80，https的缺省端口是443
通过端口可以找的对应的服务器进程
** 为什么引入端口, 而不是直接使用进程ID:
- 作为客户端无法知道服务端对应进程的 ID
- 服务端对应进程重启后 ID 会改变
- 一个网站的 web 进程 ID 是这个, 另一个网站的可能又是另一个
